Header & Logo Architecture

Guidelines for rendering the primary navigation and official logomarks.

Logo Guidelines

GreenMeans

The Logomark: The logo is rendered as a fluid, rounded circle (rounded-full) housing the official SVG (favicon-square.svg). It requires a subtle structural border (border-white/10) and a specialized drop shadow combining opacity and our core #18832b glow. Interaction should scale the inner image smoothly, not warp the container.

The Typography: The wordmark is strictly positioned alongside the logomark using the primary font in a font-light weight, with tracking-wide spacing. On hover, the wordmark should smoothly fade to the primary Green.

Header Display

Navbar Anatomy: The header operates as the structural anchor of the entire ecosystem.

• Must be a solid bg-black to maintain continuity with the main canvas.
• Requires a strict base border: border-b border-white/5.
• Elements are constrained universally by a max-w-7xl mx-auto shell.
• Flexbox is utilised (justify-between items-center) to isolate the brand quadrant (left) from contextual feature/settings actions (right).

Features & The Settings Menu

Designing feature modules, the global settings object, and interactive elements.

GreenMeans utilises a unified Settings infrastructure allowing cross-site persistence for preferences like Light/Dark mode, Reduced Motion, and Dyslexia Fonts. Elements operating within the settings module demand particular design structures emphasising legibility, concise typography, and straightforward interactive patterns. The implementation of the Settings menu must use accessible ARIA compliant modal overlays, grouping preferences logically with clear descriptive labels, and ensuring immediate, visually distinct feedback when a toggle or setting is modified.

Settings Toggle Object

All settings objects (including the main header toggle and the menu items inside the popover) consistently utilize the pill-shaped Rounded Full Architecture.

Classes: rounded-full uppercase text-xs tracking-widest text-white/70 border border-white/10 hover:border-[#18832b] hover:bg-[#18832b]/10

Hover behaviors on these components immediately swap the outline for solid #18832b backings to enforce interactivity.

Accessibility & Compliance

Guidelines for WAVE AIM compliance and inclusive design.

  • WAVE AIM Compliance

    All GreenMeans applications must strive for comprehensive WAVE (Web Accessibility Evaluation Tool) / AIM compliance. This entails rigorous adherence to semantic HTML structuring, using appropriate ARIA labels for all interactive state elements (such as settings toggles), maintaining a sequential heading structure (from H1 downward without skipping levels), and ensuring all non-text assets feature descriptive alternative text.

  • Keyboard Navigation & Focus Control

    A cornerstone of an inclusive environment is ensuring that every interactive component, particularly within the settings pane and core navigation, can be accessed seamlessly via keyboard traversal. Visible focus outlines using our primary or secondary green highlight must be robustly implemented to signify navigational progression.

  • Contrast Standards

    Contrast ratios between foreground text and underlying backgrounds must constantly meet WCAG AA standards. Our core aesthetic of bright white and light zinc atop true black safely fulfills these guidelines, making prolonged usage comfortably legible.

Language & Localisation Standards

Global editorial and linguistic styling.

  • British English Usage

    To ensure consistency and align with our core brand voice, all texts across the GreenMeans digital ecosystem—encompassing copy, documentation, interface messaging, and microcopy—must strictly employ British English spelling and formatting conventions. Thus, terms such as colour over color, utilise over utilize, and centre over center are mandatory.

Responsive Layouts & Grid Behaviours

Document width, responsive strategies, containment, and spacing parameters.

  • Container Width & Maximum Bounds

    Default max width is strictly set to max-w-7xl (1280px) for standard structural framing. For environments prioritising dense readability like documentation, drop to max-w-5xl. This ensures consistency and prevents content elements from stretching disjointedly on ultra-wide screens.

  • Mobile-First Fluidity & Margins

    Layout code is formulated mobile-first using Tailwind CSS. By default, content elements are stacked vertically. Breakpoints like sm:, md:, and lg: invoke horizontal grids or column expansions only where necessary. All layout components require a base baseline padding of px-4 or px-6 to guard text from touching device edges on handheld screens.

  • Responsive Grids

    When arranging multiple feature blocks or dashboard components, employ responsive grid utilities. A typical pattern transitions from grid-cols-1 on mobile, to md:grid-cols-2 on tablets, expanding to lg:grid-cols-3 or more on desktops. This scaling logic ensures readability and touch-target fidelity is preserved on constrained viewports.

  • Negative Space (Breathing Room)

    Generous vertical padding between sections is vital across all screen sizes. Often, an implementation will use a scaled spacer such as py-16 md:py-24 to gracefully widen the breathing room natively as screen height allows.
